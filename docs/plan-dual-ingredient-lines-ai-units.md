# Plan: dual ingredient lines + AI unit variants (conversion.py on hold)

This document describes how to pause deterministic unit conversion in `backend/app/services/conversion.py`, keep the existing HTTP surface for later, add a **second text column** for each recipe ingredient row, **backfill the alternate units with an LLM**, and **switch display and downstream behavior** based on the user’s unit preference.

---

## Goals

1. **Defer `conversion.py`** as the source of truth for ingredient-line unit math. Do not expand its rules engine or wire it into new flows until explicitly revived.
2. **Keep** `POST /units/convert` and the client helper `convertUnits` in `src/api/endpoints.ts` available for a future deterministic path (document as dormant / optional).
3. **Store two human-readable strings per ingredient row** (original + opposite-system line), generated by **AI** from the existing `line` text.
4. **Apply a single “active line” choice** (US vs metric) consistently across API responses, UI, shopping actions, and search—without rewriting the database on every toggle.

Non-goals for this phase: perfect nutritional scaling, canonical `quantity`/`unit` columns on every row, or full replacement of shopping-list dedupe logic.

---

## Current baseline (repo facts)

- **Table:** ingredient rows live in **`recipe_ingredients`**, not a table named `ingredients`. The human-readable string is **`line`** (see `backend/app/db_models.py`, `db/schema.sql`, `backend/app/recipe_db_service.py`).
- **API:** recipes expose `ingredients: string[]` and `ingredients_detail` built from the graph; `line` is the payload field in `RecipeIngredientLineCreate` (`backend/app/schemas.py`).
- **Mirror:** `recipes.ingredients_json` is a denormalized export mirror; any new column must stay consistent when rebuilding that mirror (`_write_ingredients_json_mirror`, `decode_recipe_ingredients_list`).
- **UI:** Recipe detail (`app/(drawer)/recipe/[id].tsx`) has **US / Metric** `SegmentedButtons` and imports `formatIngredientDisplay`, but **ingredient bullets still render `recipe.ingredients` verbatim**—the toggle today only affects **directions** via `formatDirectionsForDisplay`. **`convertUnits` is not called anywhere** yet—only defined in `src/api/endpoints.ts`.
- **Dedupe:** `backend/app/services/dedupe.py` still imports `convert_quantity` and `normalize_ingredient_name` from `conversion.py` for merging shopping lines. That path should be explicitly called out in this plan (keep, narrow, or replace).

## Design decisions to lock before implementation

| Decision | Options | Recommendation |
|----------|-----------|------------------|
| Column names | `line` + `line_alt` vs `line_us` + `line_metric` vs `line_primary` + `line_secondary` | Prefer **explicit system** names, e.g. **`line_us_customary`** and **`line_metric`**, or **`line`** (canonical storage) + **`line_alt_units`** with a **`line_storage_system`** enum (`us` \| `metric`) so “opposite” is unambiguous. Avoid two ambiguous `line_*` without documenting which is source. |
| Canonical row | One column is author truth; the other is derived | Treat **user-entered / imported `line` as canonical** until AI backfill; store AI output in the second column; allow manual override of either later if product requires it. |
| When AI runs | On save, on batch job, on demand | **Batch backfill** for existing DB + **on create/update** when `line` changes and alt is empty or stale (hash of `line`). |
| Toggle scope | Per screen vs persisted | Persist **user default** (already hinted in `settings.tsx`); optionally **per-recipe override** later. Minimum: **global preference** in client + pass or infer on API if server-side formatting is added. |

Document the chosen names in `db/schema.sql`, ORM, and OpenAPI-shaped schemas in one place to avoid drift between SQLite (dev) and Postgres reference schema.

## Phase 1 — Schema and migrations

1. Add nullable **`TEXT`** column(s) on **`recipe_ingredients`** per naming decision (e.g. `line_metric` if `line` is always US-customary text, or `line` + `line_alt` + `line_alt_source_system`).
2. **SQLite:** follow existing patterns (`migrate_recipe_ingredients_*`, `_sqlite_add_column_if_missing` in scripts) so local dev DBs upgrade cleanly.
3. **Postgres:** update `db/schema.sql` and any migration runner you use in deployment.
4. **ORM:** extend `RecipeIngredientRow` in `backend/app/db_models.py`.
5. **Default:** new rows may have `NULL` in the alt column until backfill or AI on save.

---

## Phase 2 — AI backfill and validation

1. **Script or admin task:** iterate all `recipe_ingredients` where alt is `NULL` (or stale), call LLM with:
   - full `line` string,
   - target system (metric vs US),
   - strict output format (single line, no commentary),
   - safety: max tokens, no ingredient invention.
2. **Validation layer (lightweight):**
   - non-empty, reasonable length,
   - optional regex heuristics (starts with digit or fraction for quantified lines),
   - reject and retry or flag row on failure.
3. **Idempotency:** store optional `line_alt_updated_at` / content hash if you need to skip unchanged rows.
4. **Cost control:** batching, rate limits, dry-run mode, logging of failures to a table or file.

Do **not** depend on `conversion.py` for this path; the LLM produces the alternate **string** directly.

---

## Phase 3 — API contract

1. **`ingredients_detail`:** include both fields per row, e.g. `line` and `line_metric` (names per Phase 1), plus optional `line_alt_status: "ok" | "pending" | "failed"` for UI.
2. **`ingredients: string[]`:** define behavior:
   - Either **deprecated** in favor of detail-only,
   - Or **computed** from the active unit system so older clients keep working (recommended during transition).
3. **Writes:** `RecipeIngredientLineCreate` (and patch flows) accept optional alt line; if omitted and `line` changed, server may enqueue AI or leave alt null per product rules.
4. **Mirror `ingredients_json`:** when rebuilding the JSON array, use the **same active line** policy as the rest of the API so exports match UI.

---

## Phase 3.5 — Future-safe contract typing (independent from AI timing)

This phase is a **safety rail**, not AI implementation work. It prevents backend/frontend drift while you iterate later on unit-conversion behavior.

1. Generate `openapi.json` from FastAPI as part of backend tooling (or CI artifact).
2. Generate TS types from OpenAPI using `openapi-typescript` into a committed file (for example `src/api/generated/openapi.d.ts`).
3. Replace manual wire contracts in `src/api/wire.ts` with aliases/wrappers derived from generated schema/path types.
4. Keep ergonomic app-level mappers if needed, but ensure canonical wire shapes come only from generated definitions.
5. Add a CI check that fails when generated API types are stale relative to backend schema.

Why it matters for your AI plan later:
- You can postpone AI conversion logic now without losing progress.
- When you later add AI-specific fields/endpoints (for example, conversion status, alternate-line provenance, confidence, or fallback strategy), frontend contracts update mechanically from backend schema.
- This reduces "silent break" risk during fast iteration on prompts, validators, and endpoint payloads.

## Phase 4 — Client: switch column everywhere

1. **Preference:** read/write global “default units” (wire from `settings.tsx` into store or persistent storage).
2. **Recipe detail:** map each ingredient row to **either** `line` or alt based on `UnitSystem`; show secondary line in muted text if you want parity with `formatIngredientDisplay` UX.
3. **New / edit recipe:** editing the canonical line should clear or mark alt stale; optionally show “Regenerate metric line” when AI is available.
4. **Shopping list add:** pass the **same string** the user sees on screen (`IngredientShoppingToggleButton` / `addRecipeIngredient`) so list items match the toggle.
5. **Search:** if search uses `ingredients` text, include **both** columns (or the active projection) so US and metric queries find the recipe.

---

## Phase 5 — `conversion.py` and `/units/convert` (on hold)

1. **Code:** add a short module docstring or `docs/` note: *deterministic conversion is paused; dual lines + AI are authoritative for display.*
2. **Endpoint:** keep `POST /units/convert` registered in `backend/app/main.py` for experiments and future merge with AI (e.g. validate LLM output or fill gaps for parsed qty/unit APIs).
3. **Client:** keep `convertUnits` in `src/api/endpoints.ts`; optionally add a one-line JSDoc: *reserved for future use — not used by ingredient list toggle.*

---

## Phase 6 — Shopping dedupe (`dedupe.py`)

Today, dedupe uses **`convert_quantity`** to merge different units on the **same normalized name**. With free-text dual lines:

- **Short term:** dedupe only when **display strings match** or when **name + unit parsing** is reliable; **disable or narrow** cross-unit merge via `convert_quantity` to avoid incorrect merges on ambiguous lines.
- **Long term:** optional second phase: parse structured qty/unit from both lines or use LLM to output normalized tuples for dedupe only.

Document the chosen behavior so shopping list merges stay predictable.

---

## Phase 7 — Testing and rollout

1. **Backend:** tests for migration, API serialization, mirror `ingredients_json` after save.
2. **Frontend:** snapshot or component tests for recipe detail with both columns populated / one missing.
3. **Rollout:** ship schema first → backfill in staging → enable UI switch when coverage threshold met (e.g. 95% rows have alt).

## Summary

Pause expanding **`conversion.py`**; keep **`/units/convert`** and **`convertUnits`** for later. Add a **second `recipe_ingredients` text column**, **fill it with AI** from the existing `line`, then **route all display and shopping-source strings** through a single unit preference that selects which column to show and export—keeping **`ingredients_json`** and **`ingredients[]`** in sync with that choice.
